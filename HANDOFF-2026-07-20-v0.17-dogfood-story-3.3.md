## 1. O que foi feito, em ordem

### 1.1 Backlog do issue #1 implementado — v0.17.0 (7 commits)

Backlog já existia commitado (`0584346`, sessão anterior) com 7 itens
priorizados por 3 passadas de análise (triagem → auditoria adversarial
Forge/GPT-5.5 → arbitragem Opus → reflect → veredito anti-overengineering).
Esta sessão implementou os 7, padrão Sonnet implementa → Opus valida
adversarialmente com prova de execução real (não só leitura de código) →
commit por fase:

- **`086f126`** — itens 1-4: fecha o bypass de tool de escrita. Matcher do
  hook trocado de `"Edit|Write|Bash"` para `"*"` + roteamento explícito em
  `main()` (Edit/Write/NotebookEdit/MultiEdit → `_evaluate_file`; PowerShell →
  `_evaluate_powershell` novo, floor-first; Bash → `_evaluate_bash`, ganhou
  paridade do floor de segredo via redirecionamento). Allowlist fixa de
  utilitárias read-only (Read/Glob/Grep/Task/WebFetch/TodoWrite). Superfície
  `docs/**` (exclui AGENTS.md/CLAUDE.md/governança). **BREAKING CHANGE**
  (matcher mudou). Validação adversarial (2 rodadas) achou e fechou: bypass
  de aspas no redirect Bash (`echo x > ".env"` escapava), MultiEdit
  blanket-denied.
- **`cf88680`** — item 5: `harness task add-file <task-id> <path>`. Append
  cirúrgico em `files[]` de uma task no `Plans.md` + recompila, sem reabrir
  `approved_by`/`approved_at`. Rejeita path com backtick/vírgula
  (corromperia o parser) — achado e corrigido na validação.
- **`1bc8ac3`** — item 6: investigação (doc oficial de hooks) confirmou que o
  `cwd` do payload `PreToolUse` é o CWD CORRENTE (não a raiz do projeto) — a
  deriva relatada no issue #1 não era só fricção, era **fail-open**
  (`_load_json` não achava `feature_list.json` sob cwd derivado → allow sem
  contrato). Fix: `repo_root` gravado em `compiled-state-session.json` na
  compilação, hook ancora a resolução de path nele. Validação achou e fechou
  falso-deny de `file_path` relativo a subdiretório; provou que traversal
  `..` não escapa da superfície.
- **`bfdcfe1`** — item 7: detecção-only de lock de arquivo (MSB3027/MSB3021/
  EBUSY/Text File Busy) em `verify_cmd` — mensagem acionável, NUNCA mata
  processo (auto-kill foi explicitamente rejeitado no backlog).
- **`c45541e`** — release 0.17.0. Suíte: 421→514.

### 1.2 Achado no próprio dogfood — v0.17.1 (`d557957`)

Ao implementar a Story 3.3 (ver 1.3), 9/9 tarefas passaram com suíte
automatizada 100% verde — mas só o teste manual em browser (pedido à parte
pelo usuário, "foram feitos os testes de UI com evidências com
screenshots?") revelou 2 defeitos reais que nenhum unit/integration/component
test pegaria: migration nunca aplicada no Oracle de dev (a design-time
factory do EF usa credenciais fictícias por design) e locale `pt-BR` nunca
registrado (formatação de moeda `R$1,000.00` em vez de `R$ 1.000,00`).

Resposta: **Passo 8 — Teste manual de UI**, regra dura nova em
`skills/plan/SKILL.md`, obrigatória sempre que alguma tarefa do contrato
tocou frontend. Depois de `harness supervise` devolver `next: null`, exige
subir a app real (nunca mock), navegar os critérios de aceitação do
`spec.md`, capturar evidência real (screenshot/DOM) e corrigir defeitos
achados antes de declarar a demanda concluída. Adicional aos `verify_cmd`
automatizados, não os substitui.

### 1.3 Dogfood real — Story 3.3 em `elegant-heisenberg`

Primeiro teste de ponta a ponta do 0.17.0/0.17.1 contra um projeto
consumidor real, story escolhida pela própria ordem recomendada no doc de
user stories do projeto (3.3 → 3.1 → 3.2). Fluxo completo: análise do
profile → entrevista → `spec.md`/`Plans.md` (9 tarefas) → aprovação →
`compile-contract --dry-run-verify` (achou e corrigiu um `cwd: frontend`
esquecido nas 4 tarefas de frontend) → `compile-session` → loop
`supervise`/`verify --mark-passed` nas 9 tarefas → regressão completa →
**teste manual real em browser** (Passo 8, motivo de ele existir) →
2 defeitos achados e corrigidos (ver 1.2) → commit `d5dcc10` em
`elegant-heisenberg`, pushed.

Resultado: 9/9 tarefas verificadas, regressão 555/555 verde (345 backend +
210 frontend), zero quebra em código pré-existente. `harness task add-file`
usado 3x durante a implementação (não estava previsto no plano original —
ver seção 2).

Confirmado em uso real: matcher `*` + `repo_root` ativos no hook instalado
(`grep` em `.claude/settings.json`/`boundary_guard.py`/
`compiled-state-session.json` do projeto-alvo), zero deny incorreto em toda
a implementação.

**Achado de ambiente, não do plugin, mas reincidente**: senha do Oracle
dessincronizada do volume Docker + migration nunca aplicada via `dotnet ef
database update` (mesma classe do item 5 rejeitado no backlog anterior —
"design-time factory usa credenciais fictícias por design"). Workaround:
`Database.MigrateAsync()` temporário em `Program.cs`, revertido após aplicar.
